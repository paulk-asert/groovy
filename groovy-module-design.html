@Module: Lightweight Module Boundaries for Groovy

Design Document · Status: exploratory, pending team review · Target release: Groovy 7 (proposed) · Author: Paul King · 24 May 2026

Purpose & status. This document sketches what an @Module annotation could look like in Groovy — what it declares, what the AST transform enforces, and how it interacts with existing visibility features. It is inspired in shape (not implementation) by Spring Modulith's application module concept, distilled down to the parts that fit a language rather than a framework. Every design decision below is proposed pending team review; alternatives are spelled out so the call can be revised in one place.

1. Motivation

Large Groovy codebases — Grails apps, Gradle plugin authoring, Spock-driven test platforms, internal monoliths — tend to grow flat package structures where any class can reach into any other class. Java has two answers: (a) package-private, too fine-grained for a logical module that spans several packages, and (b) JPMS, too coarse and too disruptive for most app-level work (separate jars, module-info.java, classpath/module-path split).

Groovy's strength is compile-time metaprogramming. An AST transform that reads a small declarative annotation and enforces it at compile time can land the “this package and these three sibling packages form a logical module, and nothing outside the module reaches into its guts” guarantee without any of the JPMS deployment machinery.

Three concrete pressures motivate this now:

  • Modular monoliths are an active architectural pattern; Spring Modulith has demonstrated demand and a working compile-time-verification model.
  • Library API protection. Groovy's own subprojects already wrestle with which classes are public surface vs internal (the @Internal / @PackageScope / @Incubating patchwork). A first-class module concept gives library authors a clearer story than “don't import from a package whose name contains .internal.”.
  • Tooling. Once modules are declared, groovydoc can render them, IDEs can warn on cross-module reaches, and architecture-rule DSLs have a target.

2. Goals & Non-Goals

2.1 Goals

  • Declare logical modules by annotating package-info.groovy — no new file types, no build-system changes required.
  • Enforce, at compile time, that one module's code only references types from modules it has explicitly declared as dependencies.
  • Enforce, at compile time, that one module's code only references the exported surface of another module — not its internals.
  • Detect cyclic dependencies between modules.
  • Be opt-in. Code without @Module behaves exactly as today.
  • Coexist cleanly with @PackageScope, @Internal, JPMS, and the existing @Incubating tier.

2.2 Non-Goals

  • Not a JPMS replacement. @Module is a source-level / logical concept; JPMS is a runtime-level / packaging concept. They can coexist.
  • Not a DI container or component scanner. Module discovery is purely lexical.
  • Not a runtime event bus, observability layer, or persistence story. Those are framework concerns (Grails, Spring Boot) and are out of scope for the language.
  • Not a full architecture-rules DSL (cycle detection lives inside the transform; richer rules are a separate, later proposal).

3. The @Module Annotation

3.1 Declaration site

The primary declaration site is package-info.groovy:

@Module(
    name = 'orders',
    allowedDependencies = ['common', 'customers'],
    exports = ['api', 'events']
)
package acme.shop.orders

The annotation applies to the package it is attached to and, by default, to every sub-package beneath it. Sub-packages may opt out or refine by declaring their own package-info.groovy with @Module (see §3.4).

3.2 Attributes

AttributeTypeRequiredMeaning
name String conditional Logical module identifier used in allowedDependencies. If omitted, defaults to the leaf segment of the package name (acme.shop.orders → orders).
allowedDependencies String[] no Logical names of modules this module may reference. Empty array means “no cross-module references allowed”. Omitted (default {}) is treated the same. See §6 for how this interacts with the JDK and external libraries (which are unrestricted).
exports String[] no Sub-package names (relative to the declaring package) whose contents form the module's public surface. If omitted, the module exports everything (back-compat / shallow adoption). See §4.
displayName String no Human-readable label for groovydoc / diagrams. Defaults to name.
onCycle Severity no One of ERROR, WARNING, IGNORE. Default proposed: ERROR; some teams may want WARNING while migrating.

3.3 Module identity

A module's identity is its name, not the package it lives in. This is deliberate: it lets a module be renamed at the package level without breaking the dependency graph, and it lets two modules in unrelated package trees reference each other by short logical names. The transform maintains a global map name → declaringPackage and rejects name collisions at compile time.

3.4 Nested modules

A package annotated @Module inside another @Module package is treated as a separate module that belongs to the same code base but is its own boundary. Cross-module rules apply between parent and child exactly as between siblings. Proposed pending review — the alternative (implicit “sub-modules” that inherit parent permissions) is more convenient but blurs the model.

4. The Exports Model

By default, every type a module declares is visible to any module that lists it in allowedDependencies. To tighten that, a module enumerates exported sub-packages:

@Module(
    name = 'orders',
    allowedDependencies = ['common'],
    exports = ['api', 'events']
)
package acme.shop.orders

Then:

  • acme.shop.orders.api.* — visible to other modules.
  • acme.shop.orders.events.* — visible to other modules.
  • acme.shop.orders.internal.* — not visible to other modules.
  • acme.shop.orders.* (the module's root) — not visible to other modules.
Default behaviour with no exports declared. Everything is exported. This makes adoption cheap: add @Module first, get dependency-direction enforcement, and only later refine to exported / internal once the API surface is understood. Proposed pending review; an alternative is “exports required, blank means nothing exported”, but that punishes incremental adoption.

4.1 Type-level overrides

For finer control without restructuring packages, a type can be explicitly marked:

  • @PublicApi — export this type even if its package is not in exports.
  • @InternalApi — do not export this type even if its package is in exports.

These compose with the package-level setting and produce a single resolved visibility per type. Proposed pending review — an alternative is to leave fine control to @PackageScope and not introduce new annotations; the counter-argument is that @PackageScope is a single-class concept and doesn't compose well with the module-wide story.

5. Compile-Time Enforcement

5.1 Phases

The AST transform runs in two phases:

  1. Discovery (per compilation unit, very early): collect every @Module declaration reachable on the source path and the compile classpath, build the module graph, and verify the graph itself (no name collisions, no cycles unless onCycle != ERROR).
  2. Reference checking (after type resolution): for every type reference in the unit being compiled, determine the source module and the target module; if they differ, verify (a) the dependency is allowed and (b) the target is exported.

5.2 What counts as a reference?

  • Imports (including import static).
  • Fully qualified type references in code.
  • Superclass / interface / generic-bound references.
  • Annotation usages.
  • Method-call receivers where the receiver type is known statically. Dynamic invocations against def typed receivers cannot be checked at compile time and are not flagged. Proposed pending review — a future @CompileStatic-only stricter mode could close this hole at the cost of surprising users in dynamic-only code.

5.3 What does not count?

  • Strings that happen to look like FQNs.
  • Class.forName(...) / reflection.
  • Test-only references when the test source set is excluded (proposed pending review — the default scope is “main source set only”, with a build-property opt-in for tests).

5.4 Error messages

Two canonical error shapes:

acme/shop/orders/OrderService.groovy: error: module 'orders' is not allowed to reference module 'billing' (no entry in @Module.allowedDependencies) reference: import acme.shop.billing.api.Invoice
acme/shop/orders/OrderService.groovy: error: type 'acme.shop.customers.internal.CustomerStore' is internal to module 'customers' and cannot be referenced from module 'orders' (package 'internal' is not in module 'customers' exports = [api, events])

5.5 Suppression

A targeted @SuppressWarnings('GroovyModuleAccess') at the field / method / class level suppresses one or more violations, mirroring the existing patterns Groovy already uses for type-checker overrides. Project-wide suppression should be done via the build, not in source.

6. Worked Examples

6.1 Minimal module declaration

// src/main/groovy/acme/shop/common/package-info.groovy
@Module(name = 'common')
package acme.shop.common

A leaf module with no dependencies and full exports. Compiles unchanged from today's behaviour.

6.2 Two modules, one-directional

// acme/shop/customers/package-info.groovy
@Module(name = 'customers', exports = ['api'])
package acme.shop.customers

// acme/shop/orders/package-info.groovy
@Module(
    name = 'orders',
    allowedDependencies = ['customers'],
    exports = ['api']
)
package acme.shop.orders

Now orders may import acme.shop.customers.api.* but not acme.shop.customers.internal.*. The reverse direction is rejected at compile time.

6.3 Type-level override

package acme.shop.orders.internal

import groovy.transform.module.PublicApi

@PublicApi
class OrderEvent { ... }   // exported despite living in an internal sub-package

6.4 External libraries and the JDK

Any reference to a type whose package is not covered by an @Module declaration is treated as “outside the module system” and is unrestricted. This means the JDK, every Groovy-stdlib package, and every third-party library work without ceremony. The model only kicks in between packages that have actively opted in.

7. Interactions with Existing Features

FeatureInteraction
@PackageScope Orthogonal. @PackageScope remains the single-class, package-private mechanism. @Module operates at a coarser grain.
@Internal (groovydoc filter) Remains a doc-only filter. A natural follow-up would be to make @Internal imply @InternalApi for module purposes, but that is proposed pending review — some users use @Internal on members of types that are themselves public surface.
@Incubating Orthogonal. A type can be exported and incubating; the two annotations describe different things (visibility vs. stability).
JPMS (module-info.java) Independent. A jar that contains @Module packages can equally well declare a JPMS module; @Module is enforced at compile time by the Groovy compiler, JPMS is enforced at runtime by the JVM. There is no required correspondence between @Module.name and the JPMS module name.
@CompileStatic / @TypeChecked The transform uses whatever type information is available. Static-typed code gets full coverage; dynamic call sites are unchecked (see §5.2).
Sealed hierarchies Complementary. Sealed restricts who can extend; module exports restrict who can reference. A sealed hierarchy that lives entirely inside an internal sub-package is a strong “closed-world” signal.
Java sources in the same compilation The Groovy compiler can read @Module annotations on Java package-info.java files for graph construction, but cannot enforce against Java source compiled by javac. Proposed pending review: ship a companion annotation processor as a follow-up for symmetry.

8. Tooling Touchpoints

Out of scope for the initial transform, but listed here so the design doesn't paint them into a corner:

  • Groovydoc. Per-module index pages, exported-type listings, and a generated dependency diagram (PlantUML or Mermaid source emitted alongside the HTML).
  • IDE integration. IntelliJ / Eclipse can read the same annotations to surface violations interactively rather than waiting for a compile.
  • Gradle / Maven. A small reporting task that dumps the discovered module graph as JSON for downstream tooling (architecture-rule plugins, dashboards).
  • Runtime introspection. A groovy.lang.Module reflective API that reports a class's declaring module is plausible but not load-bearing. Proposed pending review — defer until there is a concrete caller.

9. Implementation Phasing

Phase 1 — Annotation + dependency-direction enforcement

Ship @Module(name, allowedDependencies, displayName, onCycle), no exports yet (effectively “everything exported”). Detect missing dependencies and cycles. This is the smallest useful slice and is enough to express “these modules form a DAG”.

Phase 2 — Exports + type-level overrides

Add exports, @PublicApi, @InternalApi. This is where the “don't reach into another module's guts” story lands.

Phase 3 — Groovydoc integration

Module landing pages and dependency diagrams. Reuses the metadata Phase 1 already produces.

Phase 4 (optional) — Java APT companion

An annotation processor that enforces the same rules against Java source so a mixed-language module is symmetric. Only worth doing if mixed-language adoption shows up.

Phase 5 (optional) — Runtime reflective API

A small groovy.lang.Module facade that lets frameworks query module membership at runtime. Defer until requested by a concrete caller (Grails, a tracing library, etc.).

10. Risks & Open Questions

Risk: dynamic Groovy escape hatch

Code that uses def receivers and dynamic dispatch can sidestep the checks entirely. Mitigation: document the boundary clearly; the checks are best understood as “@CompileStatic-strength” with degraded coverage in dynamic code, not as a security boundary.

Risk: classpath-wide discovery cost

Building the module graph requires scanning package-info classes on the compile classpath. Mitigation: cache the graph per compilation; the set of declaring packages is small.

Open: should @Module imply @Internal for groovydoc?

Tempting symmetry, but @Internal is already used at member level on otherwise public types. Likely no direct implication; introduce a separate filter mode in groovydoc if needed.

Open: name vs package as identity

This document proposes logical name as identity. The alternative is to use the fully qualified package name. Logical names are friendlier; FQNs avoid a global namespace. Proposed pending review.

Open: should there be a module-info.groovy file?

JPMS-style. Rejected here for symmetry: Groovy already uses package-info.groovy for package-level annotations, and a second file type introduces parsing/build wrinkles.

Open: enforcement default for unannotated callers

Code with no @Module reaching into a module: allowed (so scripts and integration tests work) or rejected (stronger guarantee)? Proposed default: allowed, with an opt-in “closed” mode set on the module itself. Proposed pending review.

11. Reference Sketch

Annotation

package groovy.transform.module

import java.lang.annotation.*
import org.codehaus.groovy.transform.GroovyASTTransformationClass

@Retention(RetentionPolicy.RUNTIME)
@Target([ElementType.PACKAGE])
@GroovyASTTransformationClass('org.codehaus.groovy.transform.module.ModuleASTTransformation')
@interface Module {
    String name() default ''
    String[] allowedDependencies() default []
    String[] exports() default []
    String displayName() default ''
    Severity onCycle() default Severity.ERROR

    enum Severity { ERROR, WARNING, IGNORE }
}

Type-level overrides

package groovy.transform.module

@Retention(RetentionPolicy.RUNTIME)
@Target([ElementType.TYPE])
@interface PublicApi {}

@Retention(RetentionPolicy.RUNTIME)
@Target([ElementType.TYPE])
@interface InternalApi {}

Transform skeleton

// org.codehaus.groovy.transform.module.ModuleASTTransformation

@GroovyASTTransformation(phase = CompilePhase.SEMANTIC_ANALYSIS)
class ModuleASTTransformation extends AbstractASTTransformation {
    void visit(ASTNode[] nodes, SourceUnit source) {
        ModuleGraph graph = ModuleGraph.forCompilation(source)
        graph.verifyTopology(source)              // cycles, name collisions
        new ReferenceVisitor(graph, source)
            .visitModule(source.getAST())          // every type reference
    }
}

12. Summary

An @Module annotation on package-info.groovy — backed by a Groovy AST transform that verifies dependency direction and exported surface at compile time — gives Groovy users a lightweight, opt-in way to keep monoliths honest, without dragging in a framework, JPMS deployment surgery, or any runtime cost. Phase 1 alone (dependency-direction enforcement, no exports model) is small enough to land as a single GEP and is independently useful.

The design above is deliberately conservative on the language surface: one annotation, a couple of optional type-level helpers, and an AST transform. Everything richer — module-aware groovydoc, an architecture-rules DSL, runtime reflection, Java-source enforcement — is left for follow-up GEPs where each can be evaluated on its own merits.

Next step. If the broad shape is acceptable, the smallest valuable cut to prototype is Phase 1 against a real Groovy subproject (e.g. one of the GROOVY-XXXXX work areas that already wrestles with internal-vs-public boundaries) to validate the error messages and the classpath-discovery model before committing to exports.
