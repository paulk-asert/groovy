Fat-Free Lambdas Assessment

GROOVY-12043 (curryWith helpers) · GROOVY-12044 (CurriedClosure per-element allocation) · GROOVY-12054 (fat-free DGM methods) · Author: Paul King · 2 June 2026

Bottom line. The shipped fat-free idioms — the java.util.function-typed DGM overloads plus Lambdas.curryWith — run 13–23× faster than the closure / rcurry idioms on these predicate workloads and cut per-call allocation from ~72–108 bytes per element to a flat 0–24 bytes per call. The win holds in both indy and classic modes.

1. What landed, and what this measures

Three issues bring the “fat-free lambda” pattern (Don Raab, Fat-Free Lambdas in Java) to Groovy — the idea that a stateless lambda plus an auxiliary argument captures nothing, so the JVM can share one instance instead of allocating a closure per call:

IssueWhat it adds
GROOVY-12043 helpers org.apache.groovy.util.Lambdas.curryWith(...) right-partials a BiPredicate/BiFunction/BiConsumer with one value, returning a plain Predicate/Function/Consumer. Closures.curryWith(...) returns the same thing lifted into a Closure (a PredicateClosure etc.) for legacy Closure-typed APIs.
GROOVY-12044 runtime A fast path in CurriedClosure.call(Object) that skips one wrapper Object[1] for the dominant single-curried-arg shape — i.e. it trims the per-element cost of the rcurry idiom.
GROOVY-12054 DGM A java.util.function-typed overload set on the core DGM methods: any/every/find/findAll/count/collect/each/… accepting Predicate/Function/Consumer, plus @Incubating two-arg BiPredicate/BiFunction+param variants (the “With”-style overloads). The follow-up commit added count, any(BiPredicate,param), every(BiPredicate,param) and lazy collecting/findingAll.

The ad-hoc benchmark under org.apache.groovy.adhoc (originally a spike with hand-rolled stand-ins) has been rewired to call the shipped APIs and re-run. It compares eight ways (A–H) of writing the same predicate traversal, across three DGM shapes that each carry the full overload set.

2. The benchmark

Eight variants, applied to three families. The per-element work (String.startsWith) and the traversal are identical across variants; only the lambda / closure machinery differs. Prefixes are rotated and never match, so every variant performs a full traversal (find returns null, findAll an empty list, count zero).

  • C, D, G — zero, size-independent. The stateless BiPredicate (C) and method ref (D) capture nothing; the curryWith lambda (G) captures the param but never escapes find, so escape analysis removes it. count/find return a cached Integer(0)/null on no match, so even the result is free; findAll shows a constant 24 B/op — the one empty ArrayList.
  • A, B, F — linear in the element count. Each element pays for closure dispatch (A) and, for rcurry, a CurriedClosure combine (B/F). At 10000 elements that is ~0.7–1.0 MB per call.
  • H — one allocation per call, then flat. Closures.curryWith builds a single PredicateClosure (~70–130 B) and reuses it for the whole traversal — three to four orders of magnitude less than A/B at size 10000.
The allocation win does not depend on indy. C/D/G stay at 0 B/op in classic mode too — non-capturing lambdas plus escape analysis get there without the GROOVY-11905 singleton optimisation. Conversely, rcurry (B/F) allocates more under classic (~108 B/element vs ~96), so the gap widens.

  • Fat-free (C/D/G) win by ~13× over capture-closure and ~23× over rcurry, consistently across all three families and both modes.
  • G (Lambdas.curryWith) matches the direct BiPredicate+param overloads (C/D) — the curry helper is effectively free, so readability costs nothing.
  • H (Closures.curryWith) lands ~11× A — slower than the plain-functional path (it dispatches through Closure) but still a large win, and the right tool when an existing API demands a Closure.
  • Baseline note. Under indy the fat-free variants sit slightly above the naive hand-loop (E); under classic E catches up. The likely cause is the JIT inlining the monomorphic functional call into a tighter loop than the hand-written one in indy mode; the effect is small and not material to the conclusion — the practical reading is that the fat-free idioms perform at hand-loop level, while the closure idioms are an order of magnitude behind.

6. What GROOVY-12044 buys, and where the real win is

B (fresh closure + rcurry) and F (hoisted singleton closure + rcurry) allocate almost identically (~960 KB/op at size 10000), and hoisting the base closure barely moves throughput. That tells us the rcurry cost is dominated by the per-element CurriedClosure combine, not by creating the base closure. GROOVY-12044’s fast path trims that per-element cost, but the structural win — turning ~1 MB/op into ~0 — comes from not building a Closure/CurriedClosure at all, i.e. from the GROOVY-12054 functional overloads. The two changes are complementary: 12044 makes the legacy idiom cheaper for code that can’t move; 12054 lets new code skip the cost entirely.

7. CI & dashboards for the ad-hoc benchmarks

The ad-hoc benchmarks (package org.apache.groovy.adhoc) are deliberately excluded from the jmh / jmh-classic / jmh-daily suites — those filter on .bench. and .perf. patterns, which never match .adhoc.. They are exploratory, idiom-vs-idiom comparisons whose headline metric is allocation, not throughput-over-time, so folding them into the regression buckets would mix units and create alert noise.

Decision (agreed): a separate jmh-adhoc workflow with its own dashboard, leaving the existing comparison buckets untouched.

  • .github/workflows/groovy-jmh-adhoc.yml — manual (workflow_dispatch) + weekly cron; runs the \.adhoc\. pattern in indy and classic with -PjmhProfilers=gc.
  • Publishes a throughput series (jmh tool) and a separate allocation series (customSmallerIsBetter, extracted from gc.alloc.rate.norm by jmh-adhoc-alloc.py) under dev/bench/jmh-adhoc/.
  • A custom summary page (jmh-adhoc-summary.html) renders the latest run as per-variant throughput + allocation bars, with an indy/classic toggle.
  • A new opt-in -PjmhProfilers Gradle property attaches JMH profilers (default behaviour unchanged).

8. Conclusions

Fat-free idioms deliver as designed
  1. The java.util.function overloads (C/D) and Lambdas.curryWith (G) achieve zero, size-independent allocation and ~13× / ~23× the throughput of the capture-closure / rcurry idioms — at hand-loop speed.
  2. curryWith is free: G matches the direct BiPredicate+param overloads, so the readable form has no cost.
  3. Closures.curryWith (H) is the migration bridge for Closure-typed APIs: one allocation per call instead of per element, ~11× the closure idiom.
  4. The benefit is independent of indy; GROOVY-12044 helps code that must keep using rcurry, but the order-of-magnitude win is in avoiding closures entirely.
  5. Adding the count family (now that its overloads have landed) confirms the pattern generalises across element-find, list-building, and reducer shapes alike.

Suggested follow-ups proposed, pending team review

  1. Add a transform family (collect with Function / BiFunction+param and Lambdas/Closures.curryWith(BiFunction)) once a measurement design is chosen that isolates the lambda cost from the unavoidable N-element result list — e.g. report allocation net of the result, or compare against a hand-loop that builds the same list.
  2. Consider a high-hit-rate variant (prefixes that match) to show how short-circuiting find/any behave when the predicate usually succeeds.
  3. If the fat-free overloads are promoted from @Incubating, promote this benchmark family alongside and keep the weekly jmh-adhoc trend as the regression guard.
