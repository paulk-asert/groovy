groovy-contracts Graduation Assessment

Module: subprojects/groovy-contracts · Target release: Groovy 6.0.0 · Proposed JIRA: GROOVY-XXXXX: Graduate groovy-contracts from incubating to stable · Author: Paul King · 24 May 2026

Status of decisions in this document. Per-annotation graduation calls are proposed pending team review — the cleanest split (graduate the original GContracts trio + their @Repeatable containers, keep the brand-new 6.0 frame-condition / loop-variant pair on probation) is laid out below with rationale, and alternatives are spelled out so the call can be revised in one place.

1. Overview & Recommendation

The groovy-contracts module ships ten public annotations in the groovy.contracts package, nine of them currently bearing @Incubating. The module itself has been part of Groovy since the 2020 absorption of the GContracts project (GROOVY-9671); the core trio (@Requires, @Ensures, @Invariant) plus the unmarked @Contracted are five years old and stable. Three brand-new annotations land for the first time in 6.0.0: @Modifies, @ModifiesConditions, and @Decreases. The three @Repeatable container types (@RequiresConditions, @EnsuresConditions, @Invariants) were added in 2024 (GROOVY-11443) and are essentially JVM plumbing, not surface that users hand-author.

The precedent set by the two graduations earlier this week is informative: GROOVY-12030 (PropertyHandler) graduated cleanly; GROOVY-12026 (JavaShell) graduated the class but kept method-level @Incubating on two new overloads whose signatures “may evolve in a subsequent 6.x release based on usage feedback.” That mixed approach is the right template here.

Recommendation: graduate the stable seven, keep the brand-new three on probation.

Graduate now (7 annotations)

@Requires, @Ensures, @Invariant, @RequiresConditions, @EnsuresConditions, @Invariants, plus @Contracted (already not @Incubating — treat as a no-op confirmation).

Stay incubating in 6.0.0 (3 annotations)

@Modifies, @ModifiesConditions, @Decreases. Re-evaluate for 6.1 or 7.0 once there is user feedback on the frame-condition semantics, the ModifiesChecker interaction, and the loop-variant rules.

2.2 Implementation surface (out of scope for graduation; reference only)

The AST transforms in org.apache.groovy.contracts.ast (e.g. GContractsASTTransformation, ClosureExpressionEvaluationASTTransformation, LoopInvariantASTTransformation, LoopVariantASTTransformation, ModifiesASTTransformation, ModifiesEnsuresValidationTransformation) carry no @Incubating markers and aren’t part of the public API contract — they are SPI consumed via META-INF/services/org.codehaus.groovy.transform.ASTTransformation. They need no graduation action. Likewise the lifecycle and processor classes under org.apache.groovy.contracts.common are SPI; they should remain plain (or be marked @Internal, separate exercise).

The ModifiesChecker type-checking extension lives in the separate groovy-typecheckers subproject. Its graduation status is a sibling question out of scope here; note only that it is the runtime/body-level companion to @Modifies and shares its fate — if @Modifies stays incubating, ModifiesChecker likely should too.

3. Per-Annotation Decision Matrix

3.1 Graduate now recommended

AnnotationWhy graduateRisk
@Requires 5 years stable; semantics fixed (precondition, weakened by override via OR); inherited from GContracts which predated Groovy absorption. Public API surface unchanged. None foreseen.
@Ensures 5 years stable; result / old closure-parameter contract is well-established and documented. New @Modifies integration is additive (it tightens what old can reference; doesn’t change the postcondition surface itself). None foreseen.
@Invariant 5 years stable as a class invariant. Loop invariant extension via @ExtendedTarget shipped earlier (GROOVY-11878) and is documented. The annotation declaration itself is stable. Low. Loop-invariant usage on uncommon loop shapes is the main “new” surface, and it’s been in master long enough to feel done.
@Contracted Already not @Incubating. Confirm intent and add a formal user-guide section so its status is visible (currently only appears in an example). None — documentation only.
@RequiresConditions
@EnsuresConditions
@Invariants		 Pure @Repeatable container types. Users almost never write them by hand — the compiler synthesizes them from stacked annotations. Their shape is dictated by JLS rules for @Repeatable, so there is nothing left to refine. Effectively zero. The only thing that could change is which annotation they contain, and that would be a breaking change either way.

3.2 Keep incubating in 6.0.0 proposed

AnnotationWhy holdWhat would unblock graduation
@Modifies First release. Semantics carry real depth: it constrains what old may reference in a paired @Ensures, it interacts with ModifiesChecker, and the “field reference vs parameter reference” syntax (this.field vs bare name) is novel ground in the Groovy annotation vocabulary. Real users haven’t exercised it yet. (a) At least one minor release of in-field use; (b) confirmation that ModifiesChecker coverage is complete (e.g. inherited methods, indirect mutation via helpers); (c) decision on whether array element mutation should require explicit declaration; (d) richer test suite (see §5).
@ModifiesConditions Container is bound to @Modifies; graduate together. Tied to @Modifies.
@Decreases First release. The runtime contract — strictly decreasing and non-negative — is opinionated; there’s a real design question whether non-negativity should be a hard requirement or whether ordinal/Comparable termination measures (e.g. lex-ordered tuples, well-founded sets) should also be admissible. Discovering that mid-6.0 would force a breaking change. (a) Use in anger; (b) decision on whether to relax to a pure Comparable-decreasing rule with non-negativity as opt-in; (c) interaction story with structured concurrency / virtual threads (does the variant need to be thread-local?).
Alternative call for the team. If the team prefers the “graduate the whole package” approach — reasoning that @Incubating on a single module’s output is awkward to explain — the fallback is to graduate all ten and accept that @Modifies/@Decreases semantics can only be tightened (not loosened) in 6.x. The risk is concrete but bounded: the worst breaking change would be tightening the @Decreases non-negativity rule or restricting @Modifies syntax forms. I lean against this option but note it for completeness.

4. Documentation Changes

4.1 Required: user-guide bug to fix regardless of graduation outcome

Bug. subprojects/groovy-contracts/src/spec/doc/contracts-userguide.adoc line 53 lists “enabling contracts at package- or class-level with @AssertionsEnabled”. There is no @AssertionsEnabled annotation in the codebase — the actual annotation is @Contracted (checkAssertionsEnabled exists only as an internal method on Configurator). This is a leftover from the GContracts terminology and should be corrected.

4.2 Required: add a @Decreases section to the user guide

The user guide covers @Modifies at length (lines 144–247) but has no section on @Decreases at all, despite the annotation shipping. Add a section after “Loop Invariants” with: closure rules, the strictly-decreasing + non-negative contract, the LoopVariantViolation behaviour, and a worked example. Tag a code block in ContractsTest.groovy and include via tags= to match the rest of the guide. Do this even if @Decreases stays incubating — incubating doesn’t mean undocumented.

4.3 Required: add a formal @Contracted section

Currently @Contracted appears only inside the “Use within scripts”-adjacent example code. Add a short, formal section explaining: package vs class targeting, what enabling does (it’s the discovery mechanism for the AST xforms), interaction with JVM -ea/-da, and the relationship to the (mis-named) bullet currently in the feature list.

4.4 Required: javadoc @since tags on graduating annotations

Following the PropertyHandler / JavaShell precedent, add @since 2.5.0 to @Requires, @Ensures, @Invariant, @Contracted (they were first absorbed under GROOVY-9671 which targeted 2.5/3.0 era), and @since 6.0.0 to the containers if they haven’t got one already (they currently don’t). Without a @since tag the “graduated” status leaves no version anchor in the javadoc.

4.5 Optional but recommended: stability sentence in the user-guide intro

Mirror the wording used for prior graduations — one line near the top of contracts-userguide.adoc noting that the core annotations (@Requires, @Ensures, @Invariant, @Contracted) are stable from Groovy 6.0.0, with @Modifies and @Decreases remaining incubating pending feedback.

4.6 Optional: update groovy.contracts package-info.java

The current package-info enumerates only the original trio (“@Requires (precondition), @Ensures (postcondition), @Invariant (class invariant)”). Extend it to list @Contracted, mention loop invariants, and mention @Modifies / @Decreases as incubating frame-condition / loop-variant additions. This is the first javadoc page a user lands on for the package.

4.7 Optional: integration guide cross-reference

src/spec/doc/guide-integrating.adoc does not currently mention groovy-contracts. The JavaShell graduation added a section there. A short pointer (“for design-by-contract support see the contracts user guide”) would parallel that, but it’s not blocking.

5.2 No code-behaviour changes proposed

Unlike GROOVY-12008 (Sealed types) which required strengthening the transform during graduation, the contract transforms have been hardened by recent issues (GROOVY-11878 loop invariants, GROOVY-11884 annotation validation gaps, GROOVY-11885 script support, GROOVY-11894 NullChecker interplay). No additional code hardening identified as a precondition for graduation of the core trio.

6. Test Coverage

6.1 Sufficient for graduation

The original trio has solid coverage across SimplePreconditionTests, SimplePostconditionTests, ResultVariablePostconditionTests, OldVariablePostconditionTests, ResultAndOldVariablePostconditionTests, SimpleClassInvariantTests, POGOClassInvariantTests, LoopInvariantTests, InheritanceTests, the compatibility suites (CompileStaticTests, TypeCheckedTests, ImmutableTests, etc.), and the example/documentation tests. No new tests are blocking the core trio’s graduation.

6.2 Recommended additions (small)

  • One direct-use test for each @Repeatable container (@RequiresConditions / @EnsuresConditions / @Invariants) that confirms hand-written container syntax still compiles and runs the right number of assertions — the existing tests exercise them only via the synthesized path. Cheap insurance now that the containers leave incubation.
  • Following the PropertyHandlerTest precedent (133-line test added at graduation), consider a small ContractsGraduationTest in src/spec/test/ that explicitly demonstrates the stable surface — doubles as documentation.

6.3 Not required for this graduation, but flagged for the next one

If/when @Modifies and @Decreases graduate, test coverage will need to grow substantially. Today there is one ModifiesTests class and one LoopDecreasesTests class. For mature-API status the team will want: inheritance scenarios for @Modifies (does it inherit through overrides? how does it interact with @Override-via-trait?); edge cases for @Decreases (nested loops, break/continue, non-integer Comparable variants, negative-start variants).

7. Other / Out-of-Scope Notes

7.1 ModifiesChecker in groovy-typecheckers

Lives in a different subproject (groovy-typecheckers) and was added in 2026 (GROOVY-11910). Its stability is yoked to @Modifies. If the recommendation here is accepted (keep @Modifies incubating), ModifiesChecker should also remain incubating — verify and adjust during its own subproject’s graduation pass.

7.2 Container annotations as “hidden” API

Worth considering, separately from graduation: marking the three (then four) container annotations as @Internal (or noting in their javadoc that they are not intended for direct use). They’re only public because @Repeatable requires the container to be at least as visible as the contained annotation. Out of scope for this graduation but a sensible follow-up once they’re no longer incubating.

7.3 GEP / website references

The contract-related mentions on groovy-website/site/src/site/wiki are all GEP documents which serve as historical design records; no edits needed. No marketing-facing page lists the contracts module as “incubating”, so there is no banner to take down.

7.4 Module distribution / build wiring

groovy-contracts is included in settings.gradle (line 55) and is part of the standard distribution. No build-side changes are needed at graduation.
