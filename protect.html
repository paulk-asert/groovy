ASF branch-protection ruleset additions for Groovy repos

ASF Infrastructure begins submitting branch-protection PRs against all public ASF GitHub repositories on 2026-05-15. PRs auto-merge after 30 days (around 2026-06-15) if untouched. Applying the changes proactively lets the project control wording and ensure release-branch patterns are actually covered.

Current state & branch patterns

RepoRelease-branch pattern.asf.yamlRulesets?
apache/groovy GROOVY_*_X exists none
apache/groovy-geb GEB_*_X exists none
apache/groovy-website none (no version branches) missing (404) n/a

None of these repos use rel/* or release/*, so the ASF default PR would only protect each repo's default branch. Real release branches need explicit inclusion.

Recommended changes

apache/groovy

Add under the existing github: key in .asf.yaml:

  rulesets:
    - name: "Default branch protection"
      type: branch
      branches:
        includes:
          - "~DEFAULT_BRANCH"
          - "GROOVY_*_X"
      restrict_deletion: true
      restrict_force_push: true

apache/groovy-geb

Add under the existing github: key in .asf.yaml:

  rulesets:
    - name: "Default branch protection"
      type: branch
      branches:
        includes:
          - "~DEFAULT_BRANCH"
          - "GEB_*_X"
      restrict_deletion: true

apache/groovy-website

No .asf.yaml exists today — create it. Default branch is asf-site (not master); GitHub Pages is disabled, the site is served by ASF infra directly from asf-site. 

github:
  rulesets:
    - name: "Default branch protection"
      type: branch
      branches:
        includes:
          - "~DEFAULT_BRANCH"   # asf-site (published site)
          - "master"            # source branch
      restrict_deletion: true
      restrict_force_push: true

Notes

  • ~DEFAULT_BRANCH auto-resolves — no need to hardcode master/asf-site.
  • Dormant REL-BRANCH-* / REL_BRANCH_* patterns on the main repo are optional — add only for belt-and-braces on Groovy 2.x history.
  • Other Groovy repos (groovy-examples, groovy-dev-site, groovy-user-site, groovy-release) were not surveyed.
  • apache/grails-core (reference): uses <version>.x pattern (e.g. 8.0.x); their stronger layer is environments: with required reviewers for release/docs/sdkman gates — a model worth considering separately from this minimum-bar ruleset.
