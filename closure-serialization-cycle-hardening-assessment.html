Hardening Closure Deserialization Against Reference-Cycle DoS

Security assessment · Originating report: private disclosure “CurriedClosure deserialization DoS” (researcher “sonomon”) · Affects: Apache Groovy ≤ 5.0.3 (and current) · 27 June 2026

Status: Proposed — pending team review
The patch and tests below are implemented and green locally, but the decision to ship (and the precise rejection message / changelog wording) is not yet ratified by the team.

1. Summary

A security researcher reported that a hand-crafted serialized CurriedClosure whose owner/delegate point at itself triggers a StackOverflowError when deserialized-and-invoked, proposing it as a Denial-of-Service vulnerability. The PoC reuses the ysoserial toolkit and pivots through the JDK-internal AnnotationInvocationHandler gadget.

Is it a CVE?

No — consistent with the team consensus. Exploitation requires an application that deserializes untrusted Java-serialized data, which is itself the root vulnerability. The closure is merely one more gadget.

Should we harden anyway?

Yes — as defense-in-depth, if it can be done without breaking legitimate closure serialization. This assessment shows it can.

The recommended fix rejects, at deserialization time, any closure whose owner/delegate/thisObject references form a cycle through other closures — the precise condition that causes the runaway recursion. It is invisible to all legitimate closures, including curried closures and the dehydrate()/rehydrate() round-trip.

2. Root cause (confirmed)

Reproduced against groovy-5.0.6. A CurriedClosure allocated via Unsafe (bypassing the constructor) with owner = this:

  • serializes cleanly (507 bytes) and deserializes cleanly;
  • the failure is on the first invocation: CurriedClosure.call → getOwner().call → (owner is self) → call → … → StackOverflowError.

This is the same shape as Jochen’s pure-Groovy example — a cyclic owner/delegate graph that recurses forever through the MOP:

class Obj { def delegate; def invokeMethod(String n, args){ delegate."$n"(*args) } }
def o = new Obj(); o.delegate = o; o.foo()   // StackOverflowError
The leverage point is serialization, not the MOP. A depth check or visited-set in the MOP would either be a band-aid or impose a cost on the hot path; and as Jochen noted, a self-only check is trivially defeated by “one more closure in between”. The only place an attacker can build the cycle without writing Groovy is by forging a serialized stream, so that is where the cycle should be rejected.

3. Options that were ruled out

Two structural options were floated on the security thread. Both break behaviour that the test suite explicitly guarantees, so neither satisfies the “don’t break legitimate serialization” constraint.

OptionWhy it fails the constraint
(1) Remove Serializable from Closure Breaks all closure serialization, including real-world users (distributed caches/compute frameworks) and the suite’s own round-trip tests.
(2) Make owner/delegate/thisObject transient (require a rehydrated closure) They would deserialize as null. But Groovy11272.testSerialVersionUID_2/3 serialize a closure without dehydrate() and require it to be callable after the round-trip. dehydrate()/rehydrate() exist (since 1.8.5) but are opt-in; we cannot make them mandatory without breaking those tests.

4. Recommended fix

Expose a static helper Closure.checkForReferenceCycle(Closure) that runs a depth-first search over the raw owner/delegate/thisObject fields, following only Closure-valued links, and throws InvalidObjectException on any reference cycle. Groovy’s built-in serializable gadget closures — both final — call it from their own private readResolve():

// in Closure (static; not inheritable, so no impact on subclasses)
protected static void checkForReferenceCycle(Closure<?> root) throws InvalidObjectException {
    // iterative grey/black DFS over raw owner/delegate/thisObject; throws on a cycle
}

// in CurriedClosure and ComposedClosure (both final)
@Serial
private Object readResolve() throws ObjectStreamException {
    Closure.checkForReferenceCycle(this);
    return this;
}

4.1 Four design points that matter

DecisionReason
A static helper + per-class private readResolve(), not a hook on Closure itself. A hook on Closure would have to be protected to be inherited by generated subclasses, which forces every external subclass declaring the idiomatic private readResolve() (the Effective Java-recommended form) to widen its visibility and fail to compile. A static method is never inherited or overridden, so it carries no such risk; the gadget closures are final, so their private readResolve() needs no inheritance. (This is the change made in response to the Copilot review — see §6.)
Use readResolve, not readObject. A custom readObject interposes a core-loaded frame on the stack while the closure’s fields are read, shifting ObjectInputStream’s latestUserDefinedLoader() from the script’s GroovyClassLoader to the core loader — breaking deserialization of any closure whose captured field references a dynamically-generated type (caught by IntersectionClosureLiteralTest). readResolve runs after fields are read, so it does not affect class resolution.
Walk the raw fields, not the getters. CurriedClosure.getDelegate() calls getOwner().getDelegate(); using the getters on the malicious graph would itself StackOverflow. The helper lives in Closure, so it reads the private fields of every node directly, bypassing subclass overrides.
Use grey/black DFS coloring, not a plain “visited” set. A curried closure’s owner and delegate point at the same wrapped clone (a shared reference / diamond, not a cycle). A naive “revisited ⇒ cycle” rule gives a false positive on every curried closure; path-coloring distinguishes a back-edge (real cycle) from a shared reference.

The walk is iterative (explicit stack) so a very long non-cyclic closure chain cannot blow the check’s own stack. It traverses only closure-valued links, so cost is proportional to the closure graph and zero on the runtime hot path. Because readResolve throwing aborts the entire deserialization, it is sufficient for any one node on a cycle to detect it — and the outermost cycle participant (whose links are all wired by the time its readResolve runs) always does.

This answers Jochen’s objection directly: it catches arbitrary-length closure cycles (A↔B, A→B→C→A, …), not just direct self-reference.

4.2 Coverage and the opt-in for other subclasses

The check is wired into the two portable, classpath-guaranteed gadget closures (CurriedClosure — the reported one — and ComposedClosure). MethodClosure already hard-blocks deserialization (it throws unless ALLOW_RESOLVE — the CVE-2015-3253 fix; under ALLOW_RESOLVE it now also runs the cycle check). Compiler-generated and external Closure subclasses are not auto-checked, but those are not portable gadgets (an attacker needs an app-specific class they already know), and any such subclass can opt in with the same one-line private readResolve(). This is the deliberate trade-off that keeps the change fully source-compatible for existing Closure subclasses.

  • self-referential CurriedClosure → InvalidObjectException on deserialize;
  • mutual (A↔B) cycle → InvalidObjectException on deserialize;
  • self-referential ComposedClosure → InvalidObjectException on deserialize;
  • legitimate closure round-trips and remains callable;
  • legitimate curried closure round-trips and remains callable (the diamond case);
  • legitimate composed closure round-trips and remains callable.

5.3 Regression

The serialization/closure regression sweep (Groovy11272, IntersectionClosureLiteralTest, ClosureComposeTest, ClosureCurryTest, ClosureTest, ReferenceSerializationTest, MethodClosure tests) plus the new class: all green (1231 in the combined run).

The automated review raised three points; all were actioned.

7. Residual risk (by design)

This does not prevent a pure-Groovy self-inflicted infinite recursion (cl.delegate = cl; cl()). That is not a vulnerability — it is user code recursing on itself — and trying to block it in the MOP is the rabbit hole the team already declined to enter. Likewise, compiler-generated and external Closure subclasses are not auto-checked (see §4.2); they are not portable gadgets and can opt in. The hardening is scoped precisely to the attacker-controlled vector: deserialization of the built-in gadget closures.

9. Recommendation

  1. Confirm the “not a CVE” classification back to the reporter, with thanks; the hardening ships as defense-in-depth, not a security release.
  2. Adopt the static-helper design over an inheritable hook on Closure: it closes the A↔B variant, keeps the change source-compatible for all existing subclasses, and offers the same opt-in to any other serializable closure subclass.
  3. Decide the user-facing wording of the InvalidObjectException and whether a changelog note is warranted.
