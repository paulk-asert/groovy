GEP-22 Progression Analysis

Trait static members — closing the “open design point” before Groovy 6 final · GEP-22 · GROOVY-11985 · PR #2529 · 18 May 2026

Status of this document. This is a working analysis to feed the GEP-22 discussion. Every design verdict below is proposed pending team review — the JIRA thread is active and GEP-22 is not ratified. Nothing here is a decision; it is a structured restatement of the case space, the thread’s objections, and an empirical coverage map, so the team can agree definitions rather than behaviours.

1. Overview

Groovy is the only one of the four JVM trait-like systems (Java, Kotlin, Scala, Groovy) that gives each implementing class its own copy of a trait’s static members. That is what enables patterns such as overriding static defaults from the implementing class — the canonical real-world consumer being Grails’ Validateable. GEP-22 documents this but explicitly flags static-member / static-method dispatch as an open design point. The goal now is to close that point with solid definitions before Groovy 6 final.

The trigger is concrete: a silent behaviour regression between Groovy 4 and Groovy 5/6 (GROOVY-11985), discovered during the Grails 8 / Groovy 5 migration, with a fix proposed in PR #2529. The thread shows that even experienced Groovy developers disagree on what the correct behaviour is — which is precisely why principle-level definitions are needed, not another point fix.

The regression

GROOVY-8854 (Sep 2023) changed TraitReceiverTransformer so that this.staticMethod() in a trait body routes through the trait helper instead of dispatching dynamically. An implementer’s static override became silently invisible to trait code — no error, no warning.

Empirical state (verified)

Reproduced live on 6.0.0-SNAPSHOT. PR #2529 (branch groovy11985, commit 1e0b4be7de) fixes the static-method form (7/7 tests pass, dynamic + @CompileStatic). The static-field form is not fixed — characterised on the same branch.

2. The regression and the fix

2.1 What changed across versions

CallGroovy 4.0.27Groovy 5.0.5 / 6.0.0-SNAPSHOT
Direct Impl.defaultNullable()true (override)true (override)
this.defaultNullable() from trait bodytruefalse — override lost

Root cause: GROOVY-8854 rewrote the trait helper’s receiver from aload_0 (the implementing class — dynamic dispatch finds the override) to a hard-coded ldc Validateable$Trait$Helper.class (the implementing class demoted to an argument — dispatch always lands on the lowered trait default). The failure is silent and was not called out in the Groovy 5 release notes. Grails shipped a reflection workaround (Validateable.resolveDefaultNullable(Class)) that bypasses the transformer entirely.

2.2 PR #2529 mechanism

In TraitReceiverTransformer.transformMethodCallOnThis: when the resolved callee is a public, non-private static trait method, called unqualified or this.-qualified, and not inside a closure, the call is emitted against the implementing class — ((Class)$self.getClass()).m(args) tagged DO_DYNAMIC — restoring runtime dispatch to the implementer’s override. Private statics and closure context retain the old trait-helper path. It is the static-method analogue of the GROOVY-11907 field-reference fix (route through the implementer’s runtime class, mark DO_DYNAMIC), not literally the same transform.

Empirical verification. Built branch groovy11985 in an isolated worktree and ran the targeted suite: org.codehaus.groovy.transform.traitx.Groovy11985 → 7 tests, 0 failures (dynamic, @CompileStatic, instance-method-calls-static, overload-resolution-still-works). The static-method override scenario behaves like Groovy 4 again.
Scope boundary, also verified. A bare static field reference inside a trait method still binds to the trait template even on the PR branch (direct=class viaTrait=trait). PR #2529 touches method dispatch only; the static-field override path is unchanged and remains undefined. This matters directly to the Grails migration: the method-based default (defaultNullable()) is unblocked; any reliance on a static-field override is not.

3. The case space

Every disputed example in the thread is a point in an eight-dimensional space. The thread’s own tables varied only a few axes and labelled the result a “rewritten form.” The analysis below re-pivots on the capability each case exists to support, which is what GEP-22 must reason about.

  1. Member kind — static method · static field/property · (instance method, as baseline)
  2. Where declared — in trait T · only on implementer/super · both
  3. Callee visibility — public · private
  4. Call form — m() · this.m() · T.m() · external Impl.m()
  5. Call site — trait method body · inside a closure in a trait method
  6. Compilation — dynamic · @CompileStatic
  7. Inheritance — direct implementer A · B extends A · trait-static vs inherited-instance clash
  8. Override vs overload — same signature · sub-type signature (Jochen’s X1/X2)

5. Objections from the thread

Jochen Theodorou — the #1 vs #2 asymmetry.

“…this is documentation of what is the case with that PR, not what should be the case concept wise, right? … for what reason are #1 and #2 different? If the trait has the method m, then compile time checking is possible.”

Jochen — #4 standalone-compilability.

“If the method exists on A, but not T, doesn’t that mean static compilation must fail to compile T, if done without A? … This is not possible to express in bytecode and then requires a meta class path… with the big problem that the metaclass path may follow different rules than a direct call.”

Jochen — overloads, not just overrides (the X1/X2 case). Under @CompileStatic he expects 22 (the trait, compiled standalone, cannot see A); dynamic he expects 21 (A has a more specific method for X2). Groovy 4 gives 21/21, Groovy 5 gives 22/22 — “Both are imho wrong.”

Jochen — Eric’s inherited-instance case (#11). On trait T{ static m() }, class C{ def m() }, class D extends C implements T printing "T":

“In my opinion this should be a compilation error.”

Eric Milles — static implies final.

“I really wish traits never added static for its members… if you want allow defaultNullable() to be overridden, then it should not be static. Essentially static implies final.”

This is the radical option: do not support overriding trait statics at all; make redeclaration a warning/error and steer consumers (Validateable) to instance/non-static members. GEP-22 should explicitly accept or reject this, not adopt “virtual statics” by implication of merging #2529.

Jochen — private statics & the interface. Wants a hard guarantee a private trait static is callable from trait code regardless of the class; and noted the trait static method “is not added to the interface. That should be a bug.”

7. Proposed governing principles

Rather than ratify a table of behaviours, derive every row from a small principle set. Unresolvable cases are rejected only where the language already rejects them (under @CompileStatic), not by a trait-specific rule. All proposed pending team review.

PrincipleResolves
P1′Dispatch consistency, not self-containment. A name called from a trait body resolves and dispatches by the same rule whether it is declared in the trait, on the implementer, or only at runtime — and that rule matches how the same call behaves in a plain class under the same compilation mode. Dynamic traits permit missing/runtime-resolved targets exactly as dynamic classes do; @CompileStatic traits inherit the ordinary @CompileStatic resolvability requirement (no trait-specific rule, no abstract-in-T mandate). The defect to prevent is a direct-call path and a metaclass path following different rules — not the existence of dynamic resolution.Rows 4/5 (split by mode); feeds row 12 (overload/mode consistency)
P2Trait members behave like class members of the implementer. Public in-trait statics dispatch through the implementer at runtime (override wins), identically in method bodies and closures.Rows 1, 2, 6 unified (closure carve-out dropped or justified)
P3Two explicit escapes. private trait statics and T.m()-qualified calls are defined as non-virtual, trait-anchored.Rows 3, 7
P4Composition is per direct implementer. B extends A does not recompose; specified, not a bug. A trait static colliding with an inherited instance member is a compile error.Rows 10, 11
P5Typed dispatch follows the compilation mode. @CompileStatic uses the statically-resolved signature; dynamic uses MOP — consistently.Row 12

8. Decisions to resolve before Groovy 6 final

  1. The radical option (Eric). Accept or explicitly reject “static implies final — trait statics are not overridable.” Merging #2529 implicitly chooses “virtual statics”; that choice should be deliberate. Blocking — changes the public contract
  2. Static-field form (row 9). Empirically not fixed by #2529. Either extend the dynamic-through-implementer treatment to field reads, or document that static-data defaults are not overridable and Grails must use an accessor. Blocking — affects Grails migration path
  3. #1 vs #2 (closure parity). Justify the closure carve-out or remove it (P2).
  4. #4/#5. Resolved in this revision Not a trait-specific decision: dynamic = Support (works today), @CompileStatic = compile error via the ordinary resolvability rule (P1′). abstract-in-T is optional sugar only.
  5. #11. Adopt Jochen’s compile-error position or define the precedence.
  6. #12. Pick the overload rule for each compilation mode (P5).
  7. Interface exposure. Decide whether trait statics become real interface static methods (fixes external T.m(), unifies the model) or are formally erased.
  8. Tests as decisions. Every contested row needs a characterization test that encodes the chosen verdict — a dynamic positive test plus a @CompileStatic compilation-failure test for rows 4/5, and a shouldFail test for row 11, not only “works” tests.

9. Recommendation

Proposed pending team review

Merge PR #2529 to unblock the Grails 8 / Groovy 5 migration (it restores Groovy 4 semantics for the canary, row 1, with no regression in the 97k-test suite), but treat it as the first of P1′–P5, not the resolution. The PR decisively fixes row 1/10 and — verified by the matrix test — also shifts row 12 (overload) from 22 back to Groovy 4’s 21; it deliberately leaves row 2 (closures) and the row 9/11 gaps untouched. The “open design point” is the remaining rows, and they were almost entirely untested — the same gap that let the original regression ship silently, now closed by the companion matrix test.

Concretely: adopt P1′–P5 as GEP-22 text, resolve the two blocking decisions (the “static implies final” question and the static-field form) on dev@, and land a characterization test per contested row — including compile-error tests — so the definitions are enforced, not just written. Only then is the point genuinely “closed.”
