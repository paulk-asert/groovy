Groovy Properties & Configuration Roadmap

Exploring what Groovy could offer for property files and configuration — from simple format handling to a full config framework.

The Landscape Today

What Groovy Has

  • ConfigSlurper (core, groovy.util) — Groovy DSL-based configuration with environment support. Returns ConfigObject (a nested Map). Mature but has known issues: not thread-safe, no type safety, no hyphenated keys, reserved word conflicts, dot/block notation bugs.
  • ConfigObject — Map implementation with auto-creating nested access, flatten(), toProperties(), merge(), writeTo().
  • groovy-json / yaml / toml / csv — format-specific parsing including typed parseAs support, but no config-layer semantics (no merging, profiles, or expressions).

What Groovy Doesn't Have

  • Typed config binding (POJO/record from properties)
  • Multi-source merging with precedence
  • Expression evaluation (${ENV_VAR:default})
  • Hot reload
  • Standard .properties file ↔ typed object conversion
  • Thread-safe config parsing

ConfigSlurper's Known Issues

IssueDescription
Thread safetyNot safe for concurrent use. PermGen leaks reported.
No hyphenated keysCan't handle server-host style keys, incompatible with Spring Boot conventions.
Reserved word conflictsProperty named main causes MissingMethodException.
Dot vs block conflictsMixing a.b.c = x with a { b { ... } } causes errors.
Dollar sign parsing$ in property values breaks parsing.
No type safetyConfigObject is just a LinkedHashMap — no validation, no schema.
MutableNo freeze/lock mechanism; config can be modified after load.
Grails context: Grails has been working around ConfigSlurper's limitations for a decade. It built NavigableMap, PropertySourcesConfig, and dual-format merging infrastructure as band-aids — and is now deprecating those custom layers to converge on Spring Boot's configuration system. A well-designed Groovy config module could reduce the custom infrastructure both Grails and other Groovy frameworks need to maintain.

Potential Functionality — Four Layers

1 groovy-properties — Typed Properties Format Module

A new module for reading/writing .properties files with typed support, consistent with what csv/toml/yaml now offer. Pure format handling, not config management.

CapabilityApproach
Read .properties → nested MapJDK Properties + hierarchical key expansion (a.b.c → nested map). Zero dependencies.
Read .properties → typed objectJackson jackson-dataformat-properties (optional, reflective — like XML parseAs)
Write typed object → .propertiesJackson jackson-dataformat-properties
PropertiesSlurperReturns nested Map from flat properties. Supports dotted key expansion, list conventions (key.0, key.1). Zero deps.
PropertiesBuilderWrites nested structure to flat .properties format.

This gives Groovy a .properties module on par with TOML/YAML/CSV. The key innovation over java.util.Properties is hierarchical key expansion and typed binding.

Potential API:

// Zero-dep: hierarchical parsing
def props = new PropertiesSlurper().parseText('server.host=localhost\nserver.port=8080')
assert props == [server: [host: 'localhost', port: '8080']]

// Typed parsing (requires jackson-dataformat-properties)
def config = new PropertiesSlurper().parseTextAs(ServerConfig, 'server.host=localhost\nserver.port=8080')
assert config.host == 'localhost'
assert config.port == 8080

2 Format-Agnostic Config Loading

Build on top of existing format modules to load configuration from any supported format into a unified model. This is where Avaje-style functionality lives.

CapabilityApproach
Auto-detect format by extension.properties, .yml/.yaml, .toml, .json, .groovy
Load from classpath, file, URLStandard resource loading with configurable classloader
Merge multiple sourcesOrdered list with last-wins precedence
Profile/environment supportapplication-{profile}.properties convention
Expression evaluation${env.VAR:default}, ${sys.prop}, cross-references

Potential API:

def config = ConfigLoader.builder()
    .addSource('application.properties')          // classpath
    .addSource('application.yaml')                 // classpath
    .addSource("application-${profile}.yaml")      // profile-specific
    .addSystemProperties()                          // java -Dfoo=bar
    .addEnvironment()                               // env vars
    .build()

assert config.server.host == 'localhost'
assert config.server.port == '8080'

Could return a ConfigObject (backward compat), a new immutable Configuration type, or simply a nested Map.

3 Typed Config Binding

Bind a loaded configuration to typed objects — the Groovy equivalent of Spring's @ConfigurationProperties or SmallRye's @ConfigMapping.

ApproachNotes
POJO/record-based (Spring style)Bind config subtree to a class. Jackson's convertValue already does this — established pattern from XML work.
Interface-based (SmallRye style)Define interface, get implementation auto-generated. Natural fit for Groovy with AST transforms.
Groovy-native: @Immutable + bindingLeverage Groovy's existing AST transforms for immutable config objects.
Relaxed bindingserverHost, server-host, server_host, SERVER_HOST all match the same property.

Potential API:

// POJO/record binding (simplest, uses Jackson convertValue)
def serverConfig = config.getAs(ServerConfig, 'server')

// Or with an annotation-driven approach
@ConfigMapping(prefix = 'server')
record ServerConfig(String host, int port, boolean debug) {}

def config = ConfigLoader.load('application.yaml')
def server = config.bind(ServerConfig)

4 ConfigSlurper Evolution

What to do with ConfigSlurper itself:

OptionTrade-offs
A: Leave in core, deprecate nothingNo breakage, but users continue hitting known bugs.
B: Move to groovy-propertiesPackage change breaks imports. Could provide groovy.util.ConfigSlurper stub that delegates.
C: New ConfigSlurper2 in groovy-propertiesFix known issues (thread safety, hyphenated keys, reserved words). Old one stays for compat.
D: Deprecate in favour of Layer 2If the format-agnostic loader is good enough, ConfigSlurper's DSL becomes just one format option among many.
ConfigSlurper's unique value is the Groovy DSL syntax (closures, expressions, environments). If a new config loader can handle .groovy config files as one of its formats — delegating to a fixed ConfigSlurper internally — then the public API shifts to the loader and ConfigSlurper becomes an implementation detail.

Key Design Questions

  1. Should groovy-properties be purely a format module (like csv/toml/yaml), or also the home for config-loading logic? Starting as a format module and adding config-loading later — possibly as a separate groovy-config module — keeps things focused.
  2. What's the return type for config loading? ConfigObject (backward compat, mutable), a new immutable Configuration type, or just Map? ConfigObject's auto-creating behaviour (config.foo.bar creates intermediate entries on read) is convenient but also a source of bugs.
  3. Should Grails be consulted on the design? They're deprecating their custom config infrastructure and converging on Spring. They might welcome a Groovy-native config module that reduces their custom code — or they might be too far down the Spring path to benefit.
  4. Where does groovy-config end and frameworks begin? Expression evaluation, profile conventions, and source precedence are opinionated. How much should live in Groovy vs. be left to Grails/Micronaut/etc.?
