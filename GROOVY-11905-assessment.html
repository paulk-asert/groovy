GROOVY-11905 Assessment

GROOVY-11905 · PR #2438 · Optimize Non-Capturing Lambdas · Author: Daniel Sun · 16 April 2026

1. Overview

This PR optimises non-capturing lambdas in statically-compiled Groovy code. Previously, every lambda invocation allocated a new Closure wrapper object and passed it through invokedynamic to LambdaMetafactory. After this change, non-capturing lambdas (those that don’t reference this, instance fields/methods, or shared local variables) behave like Java lambdas:

  • The generated doCall method becomes static
  • The invokedynamic descriptor takes no captured arguments — ()LFunctionalInterface;
  • LambdaMetafactory can cache and return a singleton instance via ConstantCallSite

This achieves zero allocation at the call site for qualifying lambdas, matching Java’s execution model and reducing GC footprint.

2.2 New StaticTypesLambdaAnalyzer

Three nested helper classes, each with a focused responsibility:

ClassRole
InstanceMemberAccessFinder Walks the lambda body AST; sets a flag on first encounter of this/super/thisObject, instance field/method access, or qualified enclosing-instance reference (Outer.this). Short-circuits on first hit.
OuterStaticMemberResolver Builds an index of static members from the enclosing class hierarchy (including supertypes and interfaces via collectReferenceOwners()); provides findOwner() for variables, method calls, and properties.
OuterStaticMemberQualifier Transforms unqualified static references (label → Outer.label) so the static doCall can resolve them without an instance receiver.

Positive aspects:

  • Short-circuit on first instance access avoids unnecessary traversal
  • Supertype/interface walking via collectReferenceOwners() is thorough
  • Results cached via node metadata (MetaDataKey.ACCESSES_INSTANCE_MEMBERS)
  • Fresh instances per analysis call — no thread-safety issues

4. Pros

  1. Zero allocation for non-capturing lambdas: Eliminates per-invocation object creation for the most common lambda pattern, matching Java’s behaviour.
  2. Correct use of JVM infrastructure: Leverages LambdaMetafactory singleton caching and ConstantCallSite — the way the JVM was designed to be used.
  3. Clean architecture: Separation of Analyzer and Writer is a structural improvement; the GeneratedLambda record and MetaDataKey enum improve code quality.
  4. Excellent test coverage: 50+ tests covering behavioural correctness, singleton identity, capturing detection, bytecode verification, and serialisation.
  5. No compilation overhead: The analysis adds no measurable cost to compilation.
  6. No regression: All existing tests pass; capturing lambdas behave identically.
  7. Reduced GC pressure: For applications with tight loops using non-capturing lambdas (e.g., stream-style processing, functional pipelines), eliminates a source of short-lived garbage.
  8. Community endorsement: Jochen Theodorou gave +1.

5. Cons / Risks

  1. AST mutation in qualifyOuterStaticMemberReferences(): The analyser rewrites the AST in place (e.g., label → Outer.label). While this happens late in the pipeline (bytecode generation phase), it could affect IDE tooling or other AST consumers that inspect the tree afterward. A copy-on-write approach may be safer.
  2. No benchmark demonstrating the improvement: The existing JMH suite doesn’t have a benchmark specifically targeting non-capturing lambda allocation. The claimed benefit (zero allocation) is architecturally sound but unmeasured in CI.
  3. False-positive risk in capture analysis: If InstanceMemberAccessFinder misses an instance member access (e.g., through a complex expression pattern not covered by the visitor), the lambda would be incorrectly marked as non-capturing, and the static doCall would fail at runtime. The test suite is thorough, but the visitor only handles specific expression types — novel AST patterns introduced by future compiler changes could slip through.
  4. @CompileStatic only: This optimisation only applies to statically-compiled code. Dynamic Groovy code, which uses Closure rather than lambda, is unaffected. This limits practical impact to codebases using @CompileStatic with Java-style lambda syntax.
  5. Limited human review: One +1 from Jochen, one Copilot review (whose concern was addressed). More human review of the InstanceMemberAccessFinder’s completeness would be valuable — it is the critical correctness gate.
  6. isStaticMemberNamed() complexity: Property resolution checks fields, properties, and synthesised getter names (is/get prefixes). While correct for Groovy’s conventions, this multi-path resolution adds complexity that could be a source of subtle bugs if Groovy’s property conventions evolve.

6. Suggested Improvements

  1. Add a non-capturing lambda JMH benchmark. Something like: 
    @Benchmark
public int nonCapturingLambdaAllocation() {
    Function<Integer, Integer> f = (Integer x) -> x + 1;
    return f.apply(42);
}
    This would make the zero-allocation benefit visible in CI and serve as a regression test.
  2. Add a comment to qualifyOuterStaticMemberReferences() explaining that this AST mutation happens during bytecode generation and is safe because no subsequent phases read these nodes. This defends against future maintainers moving the call earlier in the pipeline.
  3. Consider a defensive verification: After marking a lambda as non-capturing and generating the static doCall, verify (perhaps as an assertion in debug builds) that the generated bytecode doesn’t reference this or instance fields. This would catch false positives during development.
  4. Document the optimisation in the Groovy user guide: A note in the lambda documentation explaining that non-capturing lambdas in @CompileStatic code are singleton-optimised (like Java) would help users understand when to prefer lambdas over closures for performance-critical code.
  5. Consider caching getOuterClasses() results: While the current uncached approach is fine for typical nesting depths, caching at the ClassNode level (with proper redirect handling, as Copilot flagged) could benefit deeply nested classes in large codebases.

7. Overall Assessment

Recommend: Approve

This is a well-designed, well-tested optimisation that brings Groovy’s lambda performance in line with Java. The code quality is high, the separation of concerns is clean, and the test coverage is excellent. The risks are manageable and the potential downsides are minor.

The lack of measurable performance improvement in the existing benchmark suite is expected — the suite doesn’t test the specific pattern being optimised. This does not diminish the value of the PR; the benefit will be visible in real-world applications that use non-capturing lambdas in @CompileStatic code, rather than in the current micro-benchmarks.

Suggested before merge: Add a dedicated lambda allocation benchmark to make the optimisation measurable in CI, and add clarifying comments around the AST mutation in the analyser.
