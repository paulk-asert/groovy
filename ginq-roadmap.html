1. Current State

GROOVY-11915 is complete. The groupby...into redesign shipped, providing first-class group variables via GroupResult<K, T>. This was identified as the foundation piece for both the SQL backend and parallel improvements.
Two platform shifts since this document began.
  • GEP-18 (Integrated Concurrency and Parallel Processing) has merged. Core Groovy now bundles native async/await/defer, an Awaitable type with an AwaitableAdapter SPI (Reactor/RxJava adapters), channels (AsyncChannel, BroadcastChannel, ChannelSelect), actors and dataflow variables, a Pool / ParallelScope abstraction, GPars-style ParallelCollectionExtensions, and Cancellable. The "future GPars module" this roadmap once assumed is now a bundled, ASL2 core runtime. GINQ's parallel story is therefore no longer "build a pool integration" but "align GINQ's bespoke pool with the bundled one" — see §4.3.
  • The SQL backend is now specified as GEP-26. The detailed dual-provider design (native + JOOQ) that previously lived in this roadmap has been promoted to a standalone proposal — see §3.

What shipped

  • GroupResult<K, T> interface extending Queryable — groups are real objects with real method calls: g.count(), g.sum(), g.toList(), g.where(), g.orderBy(), etc.
  • groupByInto on the Queryable API — clean runtime method usable from both the DSL and direct API calls.
  • Property-style key access — g.name via get(String) and g["name"] via getAt(String), replacing the originally proposed key(String).
  • having with into — having g.count() > 1 uses the group variable directly.
  • Backward compatible — classic groupby (without into) is unchanged.
  • Set operations in the DSL — union, unionall, intersect, and minus keywords combine complete from...select expressions, with left-associative chaining (Q1 union Q2 minus Q3).

Example

def result = GQ {
    from s in sales
    where s.amount > 50
    groupby s.customer as customer into g
    having g.count() > 1
    orderby g.customer
    select g.customer, g.count() as items, g.sum(s -> s.amount) as total
}

Under the covers, this transforms into fluent API calls:

from(sales)
    .where(s -> s.amount > 50)
    .groupByInto(s -> s.customer, g -> g.count() > 1)
    .orderBy(new Order(g -> g.key, true))
    .select((g, q) -> Tuple.tuple(g.key, g.count(), g.sum(s -> s.amount)))

2. Query Continuation

C# LINQ's into enables query continuation — after group...into g, you can add where, orderby, and other clauses that operate on g as a new data source:

// C# LINQ
from f in fruits
group f by f.Color into g
where g.Count() > 1
orderby g.Key
select new { g.Key, Items = g.ToList() }

The Groovy equivalent would be:

GQL {
    from f in fruits
    groupby f.color into g
    where g.count() > 1    // post-group filter (replaces 'having')
    orderby g.key
    select g.key, g.toList()
}

Current status

The where clause after groupby...into is currently rejected with a "reserved for future use" error. Users must use having instead. The having clause works fully with the group variable.

What's required

GinqExpression currently has a fixed linear structure:

from → join* → where → groupby → having → orderby → limit → select

Full continuation would require a chainable/nested model where groupby...into produces a new data source and the rest of the query operates on it — essentially a GinqExpression containing a sub-GinqExpression:

GinqExpression
├── from, join*, where, groupby (with into)
└── continuation: GinqExpression
    ├── implicit from (the group results)
    ├── where, orderby, limit
    └── select
Dependency: This could piggy-back on a general continuations feature for Groovy, which would provide the architectural substrate. Standalone implementation is moderate-to-high effort; with continuations support in the language, the GINQ-specific work would be lower.

Effort: High

This is follow-up work. The current into + having combination covers the most common use cases. Full continuation is the aspirational goal for C# LINQ parity.

3. SQL Backend

Now specified in GEP-26: GINQ SQL Backend (Native and JOOQ Providers) (Draft, target Groovy 7.0). The full design — architecture, intermediate model, expression translation, parameter binding, mixed-mode execution, module layout, effort, and phased delivery — lives in the GEP. This section keeps only a short summary so the roadmap stays self-orienting.

GINQ's syntax maps almost 1:1 to SQL, so the same query text can execute either in memory (today's collection provider) or against a relational database. GEP-26 proposes a SQL execution backend delivered as two providers sharing one front-end:

  • Native SQL provider — dependency-free, ANSI SQL via groovy.sql.Sql, fully ASL2.
  • JOOQ provider — optional, dialect-aware, window functions, via JOOQ's plain-SQL DSL.

Both consume a shared, provider-independent SqlQuery intermediate model produced once from the GINQ AST; each provider only renders the "last mile." A new high-level provider configuration key ('native-sql' / 'jooq-sql' / 'sql' auto-detect) selects the backend, sugar over the existing astWalker hook. The groupby...into redesign (GROOVY-11915) maps directly onto SQL GROUP BY + aggregates, which is what makes the walker tractable. Estimated ~2,400–3,650 lines across five phases, delivered in a new optional groovy-ginq-sql module.

GINQSQL
from e in 'employees'FROM employees e
join d in 'departments' on e.deptId == d.idINNER JOIN departments d ON e.dept_id = d.id
where e.salary > 50000WHERE e.salary > ?

orderby e.salary in descORDER BY e.salary DESC

4. Parallel & Async Processing

4.1 Current Capabilities

GINQ already has parallel support via @GQ(parallel: true) or GQ(parallel: true) { ... }:

def result = GQ(parallel: true) {
    from n in (1..1_000_000)
    where n % 2 == 0
    select n * n
}

Under the hood — and note this implementation predates GEP-18 and is entirely self-contained:

  • A dedicated ForkJoinPool ("ginq-fj-thread-N") in QueryableHelper.ThreadPoolHolder, sized by -Dgroovy.ginq.parallelism (default availableProcessors + 1)
  • A separate ExecutorService using virtual threads on JDK 21+ ("ginq-virtual-thread-N"), falling back to a fixed pool on JDK 17
  • The whole query lambda is submitted via submit(resultLambda).get() so nested parallelStream() calls inherit that pool; select/transform lambdas use QueryableHelper.supplyAsync()
  • A thread-local flag (VAR_HOLDER) carries parallel=true; QueryableCollection branches on it to pick groupingByConcurrent / toConcurrentMap

4.3 Aligning with the bundled concurrency runtime (GEP-18)

GEP-18 ships a general concurrency runtime in core Groovy. The pieces relevant to GINQ:

  • groovy.concurrent.Pool (extends Executor, AutoCloseable) with factories virtual() / fixed(n) / cpu() / io();
  • Pool.current() — the scope-bound pool set by ParallelScope.withPool(int|Pool, body); Pool.withCurrent(pool, supplier) to bind one for a body;
  • Pool.asForkJoinPool() — documented as "required for parallel stream isolation" (throws for virtual-thread pools);
  • ConcurrentConfig.getDefaultPool() / getDefaultParallelism() / -Dgroovy.concurrent.poolsize / -Dgroovy.concurrent.virtual;
  • ParallelCollectionExtensions (GPars-style eachParallel/collectParallel/…) and Cancellable.

The platform anticipated this consumer: asForkJoinPool() exists precisely for the parallel-stream isolation that GINQ relies on. So GINQ's parallelism becomes a thin adapter over the bundled runtime rather than a private pool.

Pool resolution order (the core new logic, in QueryableHelper):

  1. explicit poolSize: / pool from the query config → GINQ creates Pool.fixed(n) and owns it (closes it after, try-with-resources);
  2. else Pool.current() — an enclosing ParallelScope.withPool { GQ(parallel:true){…} } — borrowed, not closed;
  3. else ConcurrentConfig.getDefaultPool() — the shared managed default.
// Per-query pool size
def result = GQ(parallel: true, poolSize: 4) { ... }

// Share an application pool with other parallel work
ParallelScope.withPool(myPool) {
    def result = GQ(parallel: true) { ... }   // picks up Pool.current()
}

The three subtleties that are the real work (not the wiring)

  1. Virtual pools can't back parallel streams. asForkJoinPool() throws on a virtual pool, but GINQ's model is parallelStream() inside a ForkJoinPool. GINQ must resolve a FJP-backed pool (cpu()/fixed) for the stream/groupby work, using a virtual/IO pool only for the supplyAsync per-row transform path (which it already separates today). Decision needed: if a user binds a virtual pool, does GINQ reject parallel mode, fall back to a FJP for streams, or honor it only for the transform path?
  2. Scoped binding vs. InheritableThreadLocal. Pool.current() is scope-bound; GINQ's isParallel() flag is an InheritableThreadLocal (propagates on thread creation, not task submission to existing workers). Capture the pool before submit and re-establish it on the worker via Pool.withCurrent(...); audit the existing VAR_HOLDER for the same hazard.
  3. Nested-parallelism deadlock. QueryableCollection already force-materializes a hash table to "avoid nested parallel querying, which results in deadlock sometimes." Sharing one bounded pool across nested queries (or with other ParallelScope work) changes the saturation profile — verify the guard still holds; prefer FJP ManagedBlocker/work-stealing semantics.

Files touched

  • QueryableHelper.groovy — replace ThreadPoolHolder with pool resolution; submit/supplyAsync target the resolved pool; retire/rework shutdown()
  • GinqAstWalker.groovy — at the setVar(PARALLEL) + submit(resultLambda).get() site, also resolve and bind the pool (wrap in Pool.withCurrent); thread poolSize through
  • QueryableCollection.java — ensure the parallelStream() path runs under the resolved FJP (no collector-logic change)
  • GinqGroovyMethods.groovy — add poolSize (and maybe pool) to CONF_LIST + createConfiguration
  • GQ.java — optionally add a poolSize annotation member later (additive, binary compatible)
Tracking: this is the GINQ ↔ GEP-18 integration workstream — best tracked as a minor GEP / JIRA umbrella under the GINQ component, aligned to GEP-18. It is distinct from GEP-26 (SQL) and from GINQ graduation (GROOVY-12042); graduation should keep the parallel/poolSize config explicitly open to additive growth. Effort: Medium, ~300–600 lines + tests — most of it items 1–3, not the substitution.

4.4 Async Data Sources

With GEP-18 merged, the async primitives this once depended on now exist in core. Two levels:

Awaitable GINQ results — works today

def result = async {
    GQL {
        from e in employees
        where e.salary > 50000
        select e.name
    }
}
// ... do other work ...
def names = await result

No GINQ changes needed — async {} wraps any blocking call, and await is now a language feature.

Async streaming data sources — future, now buildable against real APIs

// GINQ over an AsyncChannel (groovy.concurrent.AsyncChannel now exists)
def channel = new AsyncChannel<Event>()
def alerts = GQ {
    from e in channel    // async source
    where e.severity > 3
    select e.message
}
for await (alert in alerts) {
    notify(alert)
}

This requires Queryable.from() to accept async sources and a new lazy AsyncQueryable that pulls from an AsyncChannel with backpressure. It is a genuinely new design surface, but it now builds on the merged AsyncChannel / Awaitable / for await rather than a hypothetical proposal — so it is best handled as a small GEP-18 addendum rather than a standalone GEP.
